# 数据采集工程师视角下的 TDD / DDD / 三种测试

## 概述

主流软件工程方法论（TDD、DDD）最初是为业务规则复杂、团队协作多的场景设计的。直接套用到数据采集项目（70%+ 代码在调 FFmpeg、发 HTTP、解析 HTML，外部依赖不稳定）会产生明显的"不适感"——这种不适并非工程师的问题，而是场景与方法论的不匹配。本主题回答：从 TDD/DDD 里取什么、扔什么？什么替代方案更适合数据采集？

## 分析

### 一、TDD（测试驱动开发）

**核心循环**：红（写失败测试）→ 绿（写最少代码通过）→ 重构。本质是在写代码之前逼自己把"什么叫正确"定义清楚——避免事后补测试只能证明"代码做了它做的事"，而非"代码做了它该做的事"。

**TDD 的三个隐藏前提，数据采集项目一个都不满足**：

| TDD 前提 | 数据采集的现实 |
|----------|---------------|
| 输入可精确控制 | FLV 流地址会过期、页面 HTML 会变 |
| 输出可精确断言 | 录 10 秒有时录到 9.8 秒、有时 10.2 秒 |
| 失败原因唯一 | 测试挂了是代码 bug？网络？反爬？FFmpeg 版本？ |

**结论**：纯逻辑函数（URL 解析、文件名生成、时间计算、去重、数据校验）用 TDD 很舒服；但数据采集项目里纯逻辑通常只占 20%。另外 80% 涉及 FFmpeg / HTTP / HTML 的代码不适合 TDD 循环——不是不能测，而是"能做测试但不适合 TDD"。

### 二、DDD（领域驱动设计）

**一句话定义**：用业务语言组织代码，而不是用技术语言。

```
❌ 传统（按技术层分）      ✅ DDD 风格（按业务分）
controllers/               直播间/  开播检测.py、直播间.py
services/                  采集/    流拉取.py、流切割.py
models/                    交付/    发送到COS.py、发送到Kafka.py
utils/                     查询/    历史记录.py
```

新人看到第二种结构能直接定位要改的地方——**因为文件夹名字跟开会时嘴里说的词是一样的**。

**对数据采集有用的核心概念**：

| 概念 | 大白话 | 数据采集场景 |
|------|--------|-------------|
| 实体 | 有唯一 ID、状态会变 | 直播间（ID 不变，"未开播"→"开播"） |
| 值对象 | 没 ID、不可变 | 视频切片 MP4（时长/大小/格式固定） |
| 领域事件 | 发生了什么事 | "主播 123 开播了"、"切片文件已生成" |
| 领域服务 | 不属于某对象的流程 | "从 FLV 流切成 5 分钟 MP4 片段" |

**为什么不全盘照搬**：完整版 DDD（聚合根、仓储模式、限界上下文映射）是为复杂业务规则（如订单状态机的 if-else 爆炸）设计的。数据采集的业务规则其实很薄：检测开播 → 拉流 → 切割 → 发送。过度建模反而拖慢速度。

### 三、三种测试的角色分工

**关键澄清**：外部依赖完全可以、也应该测试。问题不是"能不能测"，而是"适不适合用 TDD 循环来测"。

**单元测试 = 测一个齿轮**
- 速度 < 1ms；不联网、不读写文件、不调 FFmpeg
- 挂了说明代码逻辑写错了，定位极快
- ✅ 非常适合 TDD（输入输出完全可控）
- 示例：`generate_segment_filename(room_id, start, end) == "123456_000900_001200.mp4"`

**集成测试 = 测两个齿轮的啮合**
- 速度 1-30s；调真实 FFmpeg、读真实文件
- 挂了说明两个模块的接口对不上（输出 `.flv` 但期望 `.mp4`、文件没写完就开始读、元数据字段名不一致）
- ❌ 不适合 TDD 循环，但**必须写**——它抓的是"你以为能对接，实际对不上"的问题，恰好是数据采集最高频的 bug 类型
- 示例：FFmpeg 生成测试视频 → segmenter 切割 → 断言段数与文件存在性

**端到端测试 = 发动整车上路**
- 速度 > 30s，可能数分钟；需要本地 RTMP 服务、完整管线
- 挂了说明系统级假设不成立（拉流过程崩了、切割出空壳、上传半截中断）
- 验证的是**代码之外的环境假设**："FLV 流开播 3 秒内可拉"、"MP4 真的可播放"、"COS 上传不会断"
- 示例：本地 RTMP 推流 → 监控触发 → 全链路跑通 → 验证最终文件可播放

## 结论

### 务实做法：取需要的，扔不合适的

**从 TDD 取**：
- 纯逻辑函数写单元测试（URL 解析 / 文件名生成 / 时间计算 / 去重 / 数据校验）
- FFmpeg / HTTP / HTML 解析换集成测试，先写代码再写测试

**从 DDD 取**：
- 文件夹按业务模块组织（直播间 / 采集 / 交付 / 查询）
- 核心实体用业务语言命名（`直播间.py` 而非 `live_room_entity.py`）
- 用领域事件解耦模块（切割完成 → 事件 → 上传监听）
- ❌ 不引入 Aggregate / Repository / Bounded Context 等重概念

### 推荐项目结构

```
live-monitor/
├── 直播间监控/
│   ├── monitor.py           # 轮询 / WebSocket 检测开播
│   ├── room_resolver.py     # URL/ID 解析（← 单元测试）
│   └── platform/
│       ├── douyin.py
│       └── bilibili.py
├── 采集/
│   ├── stream_capture.py    # FFmpeg 拉流（← 集成测试）
│   ├── segmenter.py         # 切割逻辑（← 文件名单元测试）
│   └── download_task.py
├── 交付/
│   ├── cos_uploader.py
│   └── kafka_sender.py
├── shared/
│   ├── filename_utils.py    # ← 纯逻辑，单元测试
│   └── time_utils.py
└── tests/
    ├── unit/
    └── integration/
```

### 真正适合数据采集的方法论

```
防御性编程     → 每个外部调用都假设它会失败：超时、重试、降级
可观测性       → 出问题能快速定位：日志、指标、链路追踪
快速故障恢复   → 不是避免出错，而是出错了能自愈：断线重连、僵尸进程清理
```
