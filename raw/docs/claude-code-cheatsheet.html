Claude Code 使用清单 · 讲解级速查

面向读者：想重新系统梳理 Claude Code 能力、让自己更会"驾驶"而不仅仅是"使用"它的老用户。

阅读方式：每一节都按 "是什么 → 怎么用 → 为什么要这么设计 → 什么时候用 → 踩坑提醒" 的节奏展开。带颜色的小方块是重点。不用一次读完，遇到问题回来查即可。

Claude Code 的设计哲学是 "CLI 优先的 Agent 容器"：它不是一个聊天窗口，而是一个可以被你 改装 的工作台。你手上有 4 层可扩展能力——输入层（@ ! #）、会话层（记忆/权限/输出风格）、协作层（子 Agent/Skills/MCP）、自动化层（Hooks/Plugins）。理解这 4 层的分工，你就理解了 Claude Code 的全部。

1. 心智模型：Claude Code 到底是什么

一句话概括：Claude Code 是一个"会用工具的 LLM"跑在你的本地终端里。

它不同于网页版聊天的核心点是：

  • 有工具——能读写文件、跑 Bash、改 Git、上网查资料；
  • 有上下文——能把项目里的 CLAUDE.md、你当前的目录、最近的对话都作为环境；
  • 有权限边界——你可以通过 settings.json 精细控制它能做什么、该问谁；
  • 可扩展——Hooks、Skills、Subagents、MCP、Plugins 都是给你留的扩展点。

把它想象成 一个新入职的工程师，配了一个你可以自定义的"员工手册"。CLAUDE.md 是手册正文，settings.json 是权限卡，Skills 是培训课程，Hooks 是流程 SOP，Subagent 是可以临时拉来帮忙的同事。你管理它的方式，决定了它的产出。

2. 输入四剑客：@ ! # 粘贴图片

Claude Code 的输入框不是单纯的文本框，前缀字符会触发特殊模式。记住这 4 个入口，你几乎用不着打开编辑器。

2.1 @ 引用文件或目录 核心

请帮我重构 @src/utils/parser.js，让它更容易测试
对比 @src/v1/api.py 和 @src/v2/api.py，说说重写带来了什么

@ 会把文件内容注入上下文。它有两层价值：

  • 精准定位——你告诉了 Claude 要改哪里，它就不会去猜；
  • 省 token——相比让 Claude 自己 grep 找文件，你直接 @ 过去，少走几步弯路。

任何时候你 已经知道 目标文件/目录时，都用 @ 去提。只有在完全不知道从哪开始时，才让 Claude 自己搜。

2.2 ! 直接执行 Bash

! git status
! pytest tests/test_parser.py -v
! npm run build 2>&1 | tail -20

以 ! 开头的那一行直接在你的 shell 里执行，输出进入对话上下文。这一点是关键——它不是打开一个一次性终端，而是让 Claude 能"看到"命令的输出，从而在后续回答里基于这些输出做判断。

当你想让 Claude 基于某个命令的结果继续做事（比如测试失败了让它分析），先用 ! 跑一下，再让它分析，比让它自己 Bash 要快也更可控。

2.3 # 快速写入记忆

# 本项目的所有新 API 必须写 OpenAPI 注解
# 我是后端方向，前端概念需要类比 Django 的视角

# 开头的行是"记忆快捷方式"——Claude 会把这条写入 CLAUDE.md（你选一个层级：项目 / 用户全局）。下次新会话，它仍然知道。

为什么要有 #？因为 Claude 默认不会主动修改 CLAUDE.md。它希望把"记忆"动作显式交给你。# 就是一个成本极低的开关，让你不用切到编辑器也能沉淀经验。

2.4 粘贴图片 多模态

直接在输入框 Ctrl+V / Cmd+V 粘贴截图即可。Claude 能看到它。

典型场景：UI 报错截图、PO 圈出的设计稿、Excel 表格片段、手写白板。比起描述"我这个按钮挤出来了"，粘图直接解决问题。

2.5 拖拽文件（含图片）

在大多数终端里，拖文件到输入框会自动变成文件路径。如果想让 Claude 读内容，在前面加 @；如果是图片，Claude Code 会识别为图像附件。

3. 键盘快捷键全景

快捷键作用何时用
Esc中断 Claude 当前动作它走错方向了，赶紧停
Esc Esc（双击）编辑上一条消息发完才发现少说了一个关键约束
Ctrl+C退出当前会话完全离开
Ctrl+D退出 Claude Code关掉程序
Ctrl+L清屏（不清上下文）屏幕太乱但想保留对话
Ctrl+R切换到逐字展开模式（verbose）想看 Claude 每个工具调用的完整参数与输出
Shift+Tab循环切换 auto-accept / plan / 默认 模式想让它自己跑 vs 想先看计划
Tab自动补全（文件路径、命令）输入 @ 后补文件名
↑ / ↓翻命令历史重复上条指令
三个空格 + Enter多行输入模式想粘贴大段文字不误触发送
\ + Enter换行不发送同上，更简洁

Shift+Tab 是被低估的核心快捷键。它在三个模式之间循环：

  • 默认——每次危险操作都问你；
  • auto-accept（bypass permissions）——全部自动通过，适合无副作用的批量任务；
  • plan mode——先写计划不动手，你点头才执行。

养成"复杂改动前先 Shift+Tab 进 plan 模式"的习惯，能避免 80% 的返工。

4. 内置斜杠命令逐条讲

斜杠命令是 Claude Code 暴露出来的"遥控器"。下面按使用频率和重要度排序。

4.1 会话/上下文管理

命令做什么讲解
/clear 核心 清空当前对话上下文 从"任务 A"切到"任务 B"时必做。不清会导致 A 的上下文污染 B 的判断。
/compact 把长对话压缩成摘要继续 Claude Code 会在接近上下文上限时自动压缩；你也可以手动触发，加个指令告诉它"重点保留 X 相关的决策"。
/resume 恢复之前的某个会话 退出后想继续昨天那个任务，用这个挑一条。
/export 导出对话（可作为文档/复盘） 做 code review 后想把讨论沉淀成 PR 描述时好用。

很多人从头到尾不 /clear，一天下来对话里混着 5 个任务。结果就是每个回答都带着无关的前文偏见，而且 token 烧得飞快。每换一个实质性话题，就清一次。

4.2 配置与状态

/help列出所有命令。记不清时的兜底。
/status查看当前账号、模型、工作目录、已启用的 MCP 与插件。排查"怎么不对劲"的第一步。
/cost当前会话累计消耗的 token 与美元估算。
/model切换模型（Opus / Sonnet / Haiku）。
/config打开交互式配置面板：主题、输出风格、语言、快捷项。
/doctor自检：权限、网络、MCP 连通性。升级后有异常先跑它。
/upgrade升级 Claude Code 自身。
/login / /logout切换账号或重新认证。

模型选择的决策框架：

  • Opus——架构决策、复杂重构、难 bug、涉及多系统推理。慢、贵，但"想得对"。
  • Sonnet——日常编码主力。性价比最优。
  • Haiku——跑批、格式化、简单重复任务、作为 Subagent 的"打工人"。

策略：默认 Sonnet，遇到硬茬切 Opus，大量小任务并行时让 Subagent 用 Haiku。

4.3 项目管理

/init在当前项目生成一个初始 CLAUDE.md（Claude 会读代码后写摘要）。新进一个项目必跑。
/memory打开 CLAUDE.md 编辑菜单，选择要编辑的层级（项目 / 用户 / 企业）。
/add-dir把项目外的目录临时加入 Claude 可读范围。做跨仓库迁移时关键。
/permissions查看/编辑当前权限规则。

4.4 协作能力入口

/agents管理子 Agent：查看、创建、编辑。
/mcp管理 MCP 服务器：列出、启用、查看状态。
/hooks管理钩子配置。
/output-style切换输出风格（Default / Explanatory / Learning / 自定义）。
/plugin管理插件（安装/启停/升级）。

4.5 协作开发

/review对当前分支变更做 code review。相当于给自己一个"第二意见"。
/pr_comments拉取并整理某 PR 上的评论，让 Claude 协助逐条回应。
/security-review对当前改动做一次安全视角审查（SQL 注入、权限、秘密泄漏等）。

提 PR 前固定跑两条：/review + /security-review。比起等队友 review 发现问题，自己先过一遍能把扯皮降到最低。

5. 会话生命周期：clear / compact / resume / checkpoint

会话管理能力决定了 Claude Code 是"一次性工具"还是"可持续工作伙伴"。

5.1 上下文的三种状态

  1. 活跃——当前会话正在累积消息；
  2. 压缩——自动或手动触发 /compact，老消息变成摘要；
  3. 归档——你退出了，但可以 /resume 回来。

5.2 Checkpoint（回滚点）

Claude Code 会在重大编辑前自动打 checkpoint。你可以通过界面提示的 Esc Esc 回到上一条消息，配合"撤销到某个 checkpoint"恢复文件状态。

Checkpoint 和 Git 不是一回事。它是 Claude Code 内置的文件快照，为了"agent 搞砸了能拉回来"。你仍然应该勤快 git commit——checkpoint 不会跨会话保留，也不替代版本管理。

5.3 /compact 的进阶用法

/compact 重点保留：我们决定把 auth 拆成独立 service 的原因、
        以及已经改好的 3 个文件；其它讨论可以压缩。

加自然语言指令告诉 Claude 压缩的取舍。默认压缩是均匀的，指定后会偏向保留关键决策。

6. 记忆系统 CLAUDE.md 的三层结构

Claude Code 启动时会按 从大到小、从远到近 的顺序加载记忆：

层级路径适合放什么
企业策略 系统级（由管理员部署） 公司统一规范、合规要求
用户全局 ~/.claude/CLAUDE.md 你的个人偏好（如"注释写中文"、"不要用 emoji"）
项目级 <repo>/CLAUDE.md 本项目的架构、命名、常用脚本
项目本地（不提交） <repo>/CLAUDE.local.md 你个人在这个项目里的偏好，不想污染队友
子目录级 <repo>/some/sub/CLAUDE.md 只有在这个子目录工作时才加载

为什么要分层？因为不同偏好的"适用半径"不同。"我喜欢简短回答"是你的偏好，不应写进项目；"本项目用 pnpm 而不是 npm"是项目事实，不应写进个人全局。分层让规则的归属清晰，避免打架。

6.1 CLAUDE.md 写什么最值得

  1. 跑得通的命令：测试怎么跑、lint 怎么跑、dev server 怎么起。
  2. 避坑清单：过去踩过什么坑、为什么用 A 不用 B。
  3. 命名约定：文件后缀、函数前缀、文件夹语义。
  4. 外部系统指针：Linear 项目名、Grafana 面板 URL、Slack 频道。

不要把能从代码里看出来的东西抄进 CLAUDE.md（比如"这个项目用 React"）。Claude 会自己读。写那些 只能通过对话才知道 的事。

7. settings.json：权限、环境、钩子一站式配置

settings.json 是 Claude Code 的"操作系统配置"。它也分层：

企业级系统管理员部署
用户级~/.claude/settings.json
项目共享<repo>/.claude/settings.json（提交到 git）
项目本地<repo>/.claude/settings.local.json（gitignore）

7.1 权限规则：allow / ask / deny

{
  "permissions": {
    "allow": [
      "Bash(git diff:*)",
      "Bash(npm run test:*)",
      "Read(./**)",
      "Edit(./src/**)"
    ],
    "ask": [
      "Bash(git push:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Read(./.env*)",
      "Read(./**/*.key)"
    ]
  }
}

权限模式的三个段位：

  • allow——直接执行不问。
  • ask——每次问你。
  • deny——直接拒绝，Claude 连尝试都不会。

规则支持 glob。命令里用 Tool(pattern) 的语法匹配工具名+参数。

把 .env、密钥文件、rm -rf 这类写进全局 deny，一劳永逸地避免意外。项目级的 allow 里放本项目常跑的测试/构建命令，减少被打断的次数。

7.2 环境变量

{
  "env": {
    "NODE_ENV": "development",
    "ANTHROPIC_LOG": "debug"
  }
}

7.3 状态栏与输出风格

{
  "statusLine": {
    "type": "command",
    "command": "~/.claude/statusline.sh"
  },
  "outputStyle": "Learning"
}

8. Subagents 子代理：并行与上下文隔离

Subagent 是 Claude Code 里的 Claude Code——一个带独立上下文的子对话。主 agent 可以派它去做事，只把结论收回来。

8.1 为什么要有 Subagent

  1. 上下文隔离：主对话不被大量工具输出污染；
  2. 并行：可以同时派 3 个去查不同模块；
  3. 角色分工：一个专管 code review、一个专管测试、一个专管文档。

8.2 如何定义

在 ~/.claude/agents/ 或 <repo>/.claude/agents/ 放一个 Markdown 文件：

---
name: test-runner
description: 跑指定测试并精简输出。用于回归测试。
tools: Bash, Read
model: haiku
---

你是测试执行专家。收到命令后：
1. 跑指定的测试文件或测试名
2. 如果有失败，只把失败部分完整贴出来
3. 成功时用一句话总结

8.3 如何使用

Claude 会根据描述自动触发；你也可以显式说"用 test-runner agent 跑一下 auth 相关测试"。通过 /agents 命令管理。

合适的场景：① 开 3 个并行去调研不同技术方案；② 跑长时间任务（构建、测试）时不阻塞主对话；③ 有明确 SOP 的重复性活（代码审查、安全扫描、API 文档生成）。

Subagent 调用是有成本的——每派一个都会复制上下文+启动。不要把"读一个文件"这种琐事也交给 Subagent。它适合"定义清楚的、可并行的、大任务"。

9. Skills 技能：按需加载的"专业知识包"

Skill 是一个带元数据的知识/工具包，Claude 会根据 当前任务 判断是否触发。和 Subagent 的区别：

SkillSubagent
运行在哪主对话里独立子对话
目的注入专业知识/步骤承包一个独立任务
触发关键词/意图匹配显式或按 description 派遣

9.1 Skill 文件结构

.claude/skills/my-skill/
├── SKILL.md          # 主文档（Claude 会读的）
├── references/       # 详细参考（按需加载）
├── examples/         # 示例
└── scripts/          # 可执行工具

SKILL.md 的 frontmatter：

---
name: api-design
description: 设计 RESTful API 时触发。覆盖命名、错误码、版本策略。
---

# API Design

## 什么时候用这个 skill
当用户要新增 endpoint、重构 API、或讨论 API 兼容性时。

## 核心原则
...

Skill 的设计哲学叫"渐进式披露"：SKILL.md 是入口，只写最精炼的引导；细节放 references/。Claude 先读入口，需要时再读详细文件。这样在低需求场景不会浪费 token，在高需求场景又能查到完整资料。

9.2 怎么让 Skill 被可靠触发

  • description 要具体——"写 API 时用"不如"用户要设计/重构 REST endpoint 时触发"；
  • 包含关键词——"endpoint"、"route"、"API" 都写上；
  • 避开过宽——"代码审查时用"会到处触发，加限定词。

10. Hooks 钩子：让 Claude Code 自动化起来

Hooks 是 在特定事件点执行你指定命令 的机制。它让 Claude Code 从"聊天式 agent"升级为"可嵌入工作流的自动化平台"。

10.1 事件清单

事件触发时机典型用途
SessionStart会话开启加载项目上下文、检查权限
UserPromptSubmit用户发消息时预处理、关键词告警、禁止敏感词
PreToolUse工具调用前安全扫描、参数校验
PostToolUse工具调用后格式化、自动测试、更新日志
StopClaude 结束回答通知、收尾检查
SubagentStop子 agent 结束聚合结果
PreCompact压缩前把关键上下文备份出去
Notification提示时推桌面通知 / Slack

10.2 配置示例：保存后自动格式化

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\"",
            "timeout": 10
          }
        ]
      }
    ]
  }
}

10.3 用 Hook 实现"自动化行为"

关键洞察：当你说"以后每次 X 发生时都做 Y"，这必须靠 Hook 实现，不能靠 CLAUDE.md 记忆。因为 CLAUDE.md 是 建议，Claude 看心情执行；Hook 是 宿主强制执行 的，跳不过。

想稳定触发的自动行为，都用 Hook 配 settings.json。

Hook 命令在你的 shell 里跑。写脚本时注意：① 失败会中断工具执行；② 超时会被干掉；③ 不要写阻塞交互命令。开发时先从 echo 开始，验证触发点正确再替换成真实逻辑。

11. MCP 服务器：外接你自己的工具箱

MCP（Model Context Protocol）是 Anthropic 提的标准协议，让 LLM 可以连外部工具/数据源。你在 Claude Code 里看到的 mcp__chrome-devtools__*、mcp__context7__* 就是 MCP 工具。

11.1 两种部署形态

stdio本地进程，Claude Code 拉起来，通过标准输入输出通信
SSE / HTTP远程服务，Claude Code 通过 HTTP 长连接通信

11.2 配置文件位置

.mcp.json                           # 项目级，随 git 提交
~/.claude/mcp.json                  # 用户级
.claude/settings.json 里的 mcpServers # 也行

11.3 示例

{
  "mcpServers": {
    "chrome-devtools": {
      "command": "npx",
      "args": ["-y", "chrome-devtools-mcp@latest"]
    },
    "context7": {
      "command": "npx",
      "args": ["-y", "@upstash/context7-mcp@latest"]
    },
    "internal-api": {
      "command": "node",
      "args": ["./mcp-servers/internal/index.js"],
      "env": {
        "API_TOKEN": "${INTERNAL_API_TOKEN}"
      }
    }
  }
}

MCP 的甜蜜区：① 公司内部 API / 数据源（内部 SRE 看板、CRM）；② 特定领域工具（Notion、Linear、Sentry、Grafana）；③ 实时数据（chrome-devtools 看浏览器当前状态、context7 拉最新文档）。

MCP vs 插件内嵌工具 vs Hook 的边界：

  • MCP——"Claude 按需调用的工具"，像 API。
  • Hook——"事件触发时宿主跑的脚本"，Claude 不直接调用。
  • Skill——"纯知识/流程包"，不涉及外部连接。

需要 Claude 主动查询/操作外部系统 → MCP。需要某事件发生时强制执行本地动作 → Hook。

12. Plugins 插件：把能力打包分发

Plugin 是把以上这些东西（commands / agents / skills / hooks / mcpServers）打成一个可分发的包的机制。

12.1 标准结构

my-plugin/
├── .claude-plugin/
│   └── plugin.json          # 元数据
├── commands/                # 斜杠命令
│   └── my-command.md
├── agents/                  # 子 agent
│   └── my-agent.md
├── skills/                  # 技能
│   └── my-skill/SKILL.md
├── hooks/                   # 钩子
│   └── hooks.json
├── .mcp.json                # MCP 服务器
└── README.md

12.2 ${CLAUDE_PLUGIN_ROOT}

插件里引用自己的文件，一律用这个变量——这样插件被安装到任意位置都能正常工作。

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write",
      "hooks": [{
        "type": "command",
        "command": "python3 ${CLAUDE_PLUGIN_ROOT}/hooks/format.py"
      }]
    }]
  }
}

12.3 管理命令

/plugin          # 交互式管理面板
/plugin install <source>
/plugin list
/plugin enable / disable <name>

团队共用的规范、脚本、review 流程，做成插件比让每人自己 copy 好。一次发布，统一更新，版本化管理。

13. Output Styles 输出风格：改变 Claude 的"人格"

Output Style 是 Claude Code 的系统 prompt 层的可切换预设。不同风格下，Claude 的解释深度、提问习惯、注释量都不同。

13.1 内置风格

Default简洁、专业，面向有经验开发者。
Explanatory每一步都给解释，适合跨领域任务。
Learning 你当前用的主动教你原理、加学习点，适合学新框架/新概念。

13.2 自定义风格

在 ~/.claude/output-styles/ 或 <repo>/.claude/output-styles/ 放一个 Markdown 文件：

---
name: senior-reviewer
description: 严格的高级工程师视角，专挑问题
---

你是一位十年以上经验的 senior engineer 在做 review：
- 永远先指出最严重的问题
- 不写"looks good"这种客套
- 提出具体的改法或反例
- 对测试覆盖率和边界条件格外敏感

Output Style 替换的是 回答方式，不是 任务能力。它影响 Claude 怎么说话，不影响它能调哪些工具。要改能力边界，去动 Skills / Subagents / MCP。

14. 自定义斜杠命令：沉淀你的常用工作流

反复输入的长 prompt，做成斜杠命令。放在 ~/.claude/commands/（全局）或 <repo>/.claude/commands/（项目）。

14.1 基础文件格式

---
description: 给当前改动生成中文 commit message
argument-hint: [scope?]
allowed-tools: Bash(git:*)
---

请基于以下 diff 写一条中文 commit message。
格式：<type>(<scope>): <主题>

当前 staged diff：
!`git diff --staged`

未 staged 的改动（参考）：
!`git diff`

$ARGUMENTS

14.2 三种动态能力

语法作用
$ARGUMENTS用户在命令后传的全部参数
$1 $2按位置取参数
@<path>把文件内容注入
!`<cmd>`在命令定义里执行 shell，结果嵌进 prompt

做自定义命令的筛选标准：① 一周用 2 次以上；② prompt 固定 > 80%；③ 参数化能明显提速。满足两条就值得做。

15. 进阶模式

15.1 Plan Mode（计划模式）

按 Shift+Tab 切到 plan mode，Claude 会：

  1. 只读探索代码，不改任何文件；
  2. 产出一份明确的实施计划；
  3. 等你点头后退出 plan 模式执行。

涉及超过 3 个文件、架构选择、或多个合理方案的任务，先 plan 再动手。本质上是把"想清楚"和"写出来"分开，像 TDD 的红绿重构分三步一样。

15.2 Extended Thinking（长思考）

在消息里写 think / think hard / think harder / ultrathink 会分别分配递增的思考预算。遇到硬茬时加一下。

底层原理：这些关键词会让 Claude 在回答前多花 token 做"内部推理"。更深的思考在用户侧看起来就是"反应慢一点，但答案更靠谱"。不是什么时候都值得开——简单任务用反而浪费。

15.3 Headless 模式（命令行单轮）

claude -p "总结这个项目做什么" --output-format json
claude -p "修复 src/api.ts 的 lint 错误" > result.txt

-p（print）模式是非交互的一次性执行，适合脚本里嵌入 Claude、CI 流水线、定时任务。

15.4 Worktrees（隔离改动）

Git worktree 让你在同一个 repo 里开多个工作副本。Claude Code 支持在 worktree 里运行，让并行任务互不污染。

git worktree add ../my-repo-feature-x feature-x
cd ../my-repo-feature-x
claude

同时做两个无关 feature、或需要回到 main 分支跑测试又不想打断当前工作流时特别好用。

15.5 Background tasks

长耗时任务（构建、测试、爬虫）可以让 Claude 在后台跑，同时继续对话，完成后再通知你。Bash 工具的 run_in_background 参数就是这个机制。

16. IDE 集成与状态栏

16.1 IDE 扩展

  • VS Code——官方扩展，支持 inline 编辑提示、侧边栏对话；
  • JetBrains——官方插件，支持 IDEA / WebStorm / PyCharm 等。

IDE 里的优势：更直观的 diff 预览、快捷键与编辑器原生整合、文件选择器对接。

16.2 自定义状态栏

在 settings.json 里指定一个脚本，Claude Code 会周期性调用它渲染底部状态栏。

{
  "statusLine": {
    "type": "command",
    "command": "~/.claude/statusline.sh"
  }
}

脚本可以输出：当前分支、未提交文件数、时钟、token 消耗……按你想看到的来。

17. 决策树：遇到问题该拿哪把刀

17.1 我想让 Claude 在某事件发生时自动做某事

→ Hook（在 settings.json 配 PreToolUse/PostToolUse/Stop 等）。

17.2 我想让 Claude 调用某个外部系统

→ MCP 服务器（stdio 本地或 HTTP 远程）。

17.3 我有一套固定的专业知识/SOP 想让 Claude 在合适时用

→ Skill（描述清楚触发条件，把详细内容渐进披露）。

17.4 我有一个复杂的、独立的、可并行的任务

→ Subagent（定义清楚职责、工具白名单、用的模型）。

17.5 我经常输入一段相同的长 prompt

→ 自定义斜杠命令（参数化 + @ + !` 组合）。

17.6 我想改变 Claude 的说话风格

→ Output Style（自定义 prompt 模板）。

17.7 我想让 Claude 记住一些项目事实

→ CLAUDE.md（分层写：企业/用户/项目/子目录/本地）。

17.8 我想把以上一整套能力分发给队友

→ Plugin（打包发布）。

17.9 我想精细控制 Claude 能不能执行某命令

→ settings.json 的 permissions（allow / ask / deny + glob）。

17.10 我想让它在脚本/CI 里非交互地跑

→ Headless 模式 claude -p。

🎓 学习路径建议

如果你只能做 3 件事来提升使用 Claude Code 的水平，优先级：

  1. 花 20 分钟把 ~/.claude/CLAUDE.md 和当前项目的 CLAUDE.md 写好。 这是投入产出比最高的动作——你未来每次会话都在受益。
  2. 配 3 个高频自定义斜杠命令。 比如 /commit（自动生成中文 commit message）、/test-fix（跑测试并修失败）、/pr-desc（基于 diff 生成 PR 描述）。
  3. 用 Plan Mode 处理你下一个复杂改动。 体会"先想清楚再动手"和"一上来就写"的差别，你会上瘾的。

Hooks / Subagents / Skills / MCP 这些高级能力，遇到 真实痛点 时再学就够了。没有痛点硬学一遍，一周就忘光。
